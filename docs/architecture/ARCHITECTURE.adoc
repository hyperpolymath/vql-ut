== System Overview

VQL-UT (Verified Query Language — Unified Types) is a proposed dependently-typed
query language/compiler architecture for VeriSimDB. The design goal is to cover
10 progressive safety properties in a single system. The current repository is
an early prototype and design vehicle, not a finished compiler that already
produces machine-checkable proofs end to end.

In the intended architecture, every query entering VQL-UT would pass through a
six-stage pipeline. Each stage would be responsible for enforcing one or more
of the 10 safety levels. Proof-certificate output remains a design target, not
an implemented release guarantee in the current repo.

== The 10 Safety Levels

VQL-UT enforces 10 distinct safety levels, each targeting a specific class of
bug. The levels are cumulative — a query at Level N has passed all checks from
Levels 1 through N.

[cols="1,2,3,2,2"]
|===
|Level |Name |What It Prevents |Idris2 Module |Pipeline Stage

|L1
|Construction Safety
|SQL/VQL injection attacks
|`Injection.idr`
|Parse + Bind

|L2
|Schema Pinning
|Schema drift (query vs database mismatch)
|`SchemaEvol.idr`
|Bind

|L3
|Resource Linearity
|Connection leaks, double-use bugs
|`Linear.idr`
|Check

|L4
|Session Protocols
|Use-after-commit, auth bypass, double-close
|`Session.idr`
|Check

|L5
|Effect Tracking
|Undeclared side effects (writes in read-only context)
|`Effects.idr`
|Check

|L6
|Scope Isolation
|Cross-transaction data leaks
|`Modal.idr`
|Check

|L7
|Information Flow
|Security classification violations (secret → public)
|`InfoFlow.idr`
|Check + Analyse

|L8
|Quantitative Bounds
|Resource overuse beyond declared limits
|`Quantitative.idr`
|Check

|L9
|Proof Attachment
|Unverified claims about query results
|`ProofCarrying.idr`
|Analyse

|L10
|Cross-Cutting Invariants
|Violations of compositional properties
|`Proofs.idr`
|Analyse
|===

=== Level Detail

==== L1: Construction Safety (Injection.idr)

Queries are unforgeable by construction. The `SafeQuery` type is an opaque AST
indexed by a parameter schema. There is no function `String -> SafeQuery` — user
input can only appear in typed parameter slots, never in query structure
positions. Injection is impossible because the path from untrusted input to
query structure simply does not exist in the type system.

==== L2: Schema Pinning (SchemaEvol.idr)

Every query is indexed by the schema version it was validated against. A
`SchemaQuery v` can only execute against a database at version `v`. Schema
migrations produce `Migration from to` proofs, and queries must be explicitly
re-validated via `migrateQuery` with evidence that all referenced columns
survive the migration.

==== L3: Resource Linearity (Linear.idr)

Database connections carry a usage count at the type level. A `LinConn (S n)`
must be consumed exactly once per scope — the Idris2 compiler enforces this
via QTT quantity `1`. Dropped connections are compile-time errors. Double-use
is impossible.

==== L4: Session Protocols (Session.idr)

Sessions are state machines indexed by their protocol state. The type
`Session : SessionState -> Type` ensures transitions follow the protocol:
`Fresh → Authenticated → InTransaction → Committed → Closed`. Querying after
commit, committing without a transaction, and closing twice are all type errors.

==== L5: Effect Tracking (Effects.idr)

Each query declares its effects (`Read`, `Write`, `Cite`, `Audit`, `Transform`,
`Federate`). The `Subsumes` proof verifies that actual operations are a subset
of declared effects. A query declared `EFFECTS { Read }` that attempts a write
is rejected at compile time.

==== L6: Scope Isolation (Modal.idr)

Data accessed in one transaction scope is wrapped in `Box w a`, where `w` is a
world tag representing the scope. Extracting data requires proof of being in the
same scope (`InScope w`). Cross-scope data flow requires explicit `marshal`
calls, making all boundary crossings visible in the type signature.

==== L7: Information Flow (InfoFlow.idr)

Data carries security classifications (`Public`, `Internal`, `Confidential`,
`Secret`, `TopSecret`) forming a lattice. The `FlowsTo` relation ensures data
can only move upward in classification. Reading a `Secret` value requires
`HasClearance Secret` evidence and a `FlowsTo Secret readerLevel` proof.
Downward flows are impossible — the constructor does not exist.

==== L8: Quantitative Bounds (Quantitative.idr)

Generalises linearity from "exactly once" to "at most N". A
`BoundedResource n a` can be consumed `n` times before depletion. Parallel
query branches can split a resource budget via `merge`/`split`. This enforces
API rate limits, connection pool bounds, and quota compliance at compile time.

==== L9: Proof Attachment (ProofCarrying.idr)

Query results carry formal theorems as dependent pairs:
`ProvedResult a thm` bundles data with a proof of integrity, freshness,
provenance, or consistency. The proof can be independently verified via
`verify`, which returns `Either VerifyError (result : a ** Holds thm result)`.

==== L10: Cross-Cutting Invariants (Proofs.idr)

Compositional properties that span multiple levels: subset transitivity,
subsumption monotonicity, flow lattice transitivity. These proofs ensure that
the safety levels compose correctly — e.g., adding effects never breaks
existing subsumption proofs, and chained migrations preserve column survival.

== Technology Stack

[cols="1,2,3"]
|===
|Technology |Role |Rationale

|**Idris2**
|ABI definitions, type kernel, proof obligations
|Native QTT (quantity `0`/`1`/`ω`) makes linear types and resource tracking
first-class. Dependent types enable schema-indexed queries. `%default total`
with zero `believe_me` guarantees all proofs are real.

|**Zig**
|FFI bridge, native codegen
|C-ABI compatible without overhead. Cross-compilation built-in. Zero runtime
dependencies. Exports the type checker's validation functions to non-Idris2
consumers via the hyperpolymath ABI/FFI standard.

|**ReScript**
|Surface syntax parser
|Consistent with existing VQL parser patterns in VeriSimDB. Compiles to
JavaScript for browser-based tooling. Extends the VQL v3.0 grammar with 6
new optional clauses.

|**V-lang**
|HTTP API gateway
|Exposes VQL-UT as a REST API for external consumers. Fast compilation,
minimal runtime. Follows the hyperpolymath V-lang API triple pattern.

|**Elixir**
|VeriSimDB integration, query orchestration
|BEAM runtime for distributed query dispatch. Phoenix for real-time feedback.
Zig NIFs for performance-critical paths. Native supervision trees for fault
tolerance.
|===

== The TypeLL Connection

VQL-UT's 10 safety levels map 1:1 to TypeLL's 10 type theory levels. This is
not a coincidence — VQL-UT is the *database specialisation* of TypeLL's
general-purpose type system.

[cols="1,2,2,2"]
|===
|Level |TypeLL Concept |VQL-UT Instantiation |Module

|1
|Construction Types
|Unforgeable query ASTs
|`Injection.idr`

|2
|Indexed Types
|Schema-versioned queries
|`SchemaEvol.idr`

|3
|Linear Types
|Resource-counted connections
|`Linear.idr`

|4
|Session Types
|Database protocol state machines
|`Session.idr`

|5
|Effect Types
|Declared vs actual effect sets
|`Effects.idr`

|6
|Modal Types
|Transaction scope isolation
|`Modal.idr`

|7
|Information Flow Types
|Security classification lattice
|`InfoFlow.idr`

|8
|Quantitative Types
|Bounded resource accounting
|`Quantitative.idr`

|9
|Proof-Relevant Types
|Theorem-carrying query results
|`ProofCarrying.idr`

|10
|Metatheoretic Types
|Cross-cutting compositional invariants
|`Proofs.idr`
|===

TypeLL provides the *theory*; VQL-UT provides the *practice*. Every TypeLL level
has a concrete, tested implementation in VQL-UT's Idris2 kernel. This makes
VQL-UT both a useful tool and a validation of TypeLL's design.

== Error Propagation

Each safety level produces typed errors that propagate through the pipeline:

[cols="1,2,3"]
|===
|Level |Error Type |Example

|L1
|`InjectionError`
|"Untrusted string in query structure position"

|L2
|`SchemaMismatch`
|"Column 'age' does not exist at schema version 2"

|L3
|`LinearityViolation`
|"Connection used twice in linear context"

|L4
|`ProtocolViolation`
|"Cannot query in state Committed"

|L5
|`EffectViolation`
|"Write effect not declared (declared: [Read])"

|L6
|`ScopeEscape`
|"Data from transaction scope 'tx1' leaked to 'tx2'"

|L7
|`ClassificationViolation`
|"Secret data cannot flow to Public context"

|L8
|`ResourceExhausted`
|"Resource limit 5 exceeded (attempted 6 uses)"

|L9
|`ProofFailure`
|"IntegrityTheorem could not be discharged"

|L10
|`CompositionError`
|"Subsumption monotonicity violated after effect addition"
|===

Errors are *structured*, not string messages. Each error carries the evidence
needed to understand and fix the problem — including the safety level, the
specific obligation that failed, and the source location.

== The Proof Certificate Concept

A proof certificate is a machine-checkable artefact produced by VQL-UT's
pipeline. It attests that a query has passed all applicable safety levels.

Properties of a proof certificate:

1. **Independently verifiable**: Any party with the Idris2 type kernel can
   re-check the certificate without trusting VQL-UT's implementation.

2. **Tamper-evident**: The certificate includes a SHA-256 hash of the query
   AST. Modifying the query invalidates the certificate.

3. **Level-granular**: The certificate reports each safety level individually.
   A query might pass L1–L8 but fail L9 (proof attachment). This granularity
   enables graduated deployment.

4. **Composable**: Certificates for sub-queries compose into certificates for
   compound queries, subject to L10 (cross-cutting invariant) checks.

5. **Versioned**: Certificates are pinned to schema version, Idris2 kernel
   version, and VQL-UT grammar version. Version mismatches require
   re-certification.
